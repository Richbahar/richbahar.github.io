首页 Human Interface Guidelines

模态情景是一种设计技术，它以一种临时模式呈现内容，这种模式与用户先前的当前上下文分离，并且需要一个明确的退出操作。

# 模态情景

模态情景可以

  • 帮助用户专注于一个独立的操作或一组紧密相关的选项
  • 确保人们收到重要信息，并在必要时采取行动

iOS 提供了警报、活动视图 (或共享表) 和动作表，特定情况下你可以在 APP 使用它们。为了在 APP 中呈现自定义的模态情景内容，iOS 13 及以后版本支持以下显示样式。

# 页面表

页面表显示为一张小卡片，其中的卡片部分覆盖了基础内容，未覆盖的区域变暗防止用户点击。主视图的顶部边缘或先前的卡在当前的卡后面是可见的，使用户记住遇到弹出提示前的操作。人们通过以下方式关闭弹窗提示：

  • 从屏幕顶部向下滑动
  • 卡内容滚动到顶部时，可以从屏幕上的任意位置向下滑动
  • 点击一个按钮

为不启用复杂任务的非浸入式场景使用一个页面表。

# 全屏显示

全屏展示覆盖了整个屏幕。以前的页面看不到，最大程度地减少视觉干扰。用户通过点击按钮来关闭全屏模式视图。

使用全屏模式视图获取沉浸式内容（例如视频，照片或照相机视图），或用在全屏演示效果更好的复杂任务（例如标记文档或编辑照片）。

如果使用当前上下文模式视图样式拆分视图窗格、弹出窗口或其他非全屏视图中显示模式内容，则在紧凑环境中显示模式内容时，应切换为使用工作表。

# 在有意义时使用模态

仅在将人们的注意力集中在做出选择或执行与当前操作不同的任务上关键时期，才创建模态体验。模态体验会使人们脱离当前环境，需要采取行动才能关闭，因此只有在提供明显好处时才使用它。

# 保留提供必要且理想的可操作的提示信息

通常，出现问题时需要提示用户。由于提示会中断体验并需要点击才能关闭，因此让用户感到这个提示是必要的，这一点很重要。有关指导，请参阅提示。

# 保持模态任务简单，简短，重点突出

不要在您的 APP 中创建 APP。如果模态任务过于复杂，⼈们可能会在进⼊模态上下文时忽略他们之前的操作。尤其要谨慎创建涉及视图层次结构的模态任务，因为用户可能忘记操作到哪里了并忘记如何回到之前的操作路径。如果模态任务必须包含子视图，请提供层次结构中的单个路径以及完成的明确路径。避免使用 “完成” 按钮执⾏除完成任务之外的其他操作。

# 提供⼀种明显且安全的退出模态任务的按钮

例如，您可以使用 “完成” 或 “取消”。确保退出按钮可以正常使用，并提供了一个替代关闭动作的手势。

# 如果适⽤，显示标识任务的标题

当人们进入模态任务时，他们会离开先前的页面，因此最好将新的页面内容弄清楚。您也可以在视图的其他部分提供文本，更完整地描述任务或提供指导。

# 模态视图与 APP 设计保持一致

例如，当模式视图包含导航栏时，其外观应与 APP 中的导航栏相同。

# 使用在 APP 有意义的模态过渡样式

使用与 APP 相协调的过渡样式，并增强对临时内容转移的意识。默认过渡是在模式视图关闭时，默认过渡从屏幕底部垂直向上滑动，并在关闭时返回。在整个应用中使用一致的模式过渡样式。

有关开发人员指南，UIView 控制器和 UIPresentation 控制器

# 导航

用户往往不会意识到 APP 的导航，直到它无法满足需求。您的工作就是以一种支持 APP 结构和目的的方式来设计导航，而无需关注导航自身。导航应该自然而又熟悉，并且不应该主导界面或将焦点从内容移开。在 iOS 中，有三种主要的导航样式。

# 分层导航

每个页面作出一个选项，直到到达需求页面。要到达另一个页面，您必须返回或从头开始并做出不同的选择。设置和邮件使用此导航样式。

# 平面导航

在多个内容类别之间切换。音乐和 App Store 使用此导航样式。

# 内容驱动或体验驱动的导航

在内容之间自由移动，或者内容本身定义导航。游戏，书籍和其他沉浸式应用通常使用这种导航样式。

有些 APP 结合了多种导航样式。例如，使用平面导航的 APP 可以在每个类别内实施分层导航。

# 始终提供清晰的路径

用户应该始终知道他们在 APP 中的位置以及如何到达下一个目的地。无论导航风格如何，内容的路径必须合理，可预测且易于遵循。通常，给人们一个通往每个页面的路径。如果他们需要在多个上下文中查看页面，请考虑使用操作表，警报，弹出窗口或模式视图。要了解更多信息，请参阅操作表，警报，弹出窗口和模态视图。

# 设计一种信息结构，使用户快速而轻松地获取内容

以最少的点击，滑动和显示屏幕的方式来组织信息结构。

# 使用触摸手势创建流畅性

使用最小摩擦力跳转界面。例如，您可以让人们从屏幕侧面滑动以返回上一个屏幕。

# 使用标准导航组件

尽可能使用标准导航控件，例如页面控件，选项卡栏，分段控件，表视图，集合视图和拆分视图。用户已经熟悉这些控件，并且会直观地知道如何使用您的应用程序。

# 使用导航栏遍历数据层次结构

导航栏的标题可以显示层次结构中的当前位置，并且使用后退按钮可以轻松返回到先前的位置。有关具体指导，请参见导航栏。

# 使用选项卡栏显示内容或功能的同级类别

标签栏使人们可以快速轻松地在类别之间进行切换，而与当前位置无关。有关具体指导，请参见标签栏。

# 在 iPad 上，使用拆分视图而不是标签栏

拆分视图提供与选项卡栏相同的快速导航，同时更好地利用了大屏幕。有关指导，请参见拆分视图。

# 当您具有内容类型相同的多个页面时，请使用页面控件

页面控件清楚地传达了可用页面的数量以及当前处于活动状态的页面。天气应用程序使用页面控件来显示特定于位置的天气页面。有关具体指导，请参见页面控件。

# 请求权限

用户必须授予 APP 访问个人信息的权限，包括当前位置，日历，联系信息，提醒和照片。尽管用户喜欢这带来的便利，但他们也希望能够保护隐私数据。例如，用户喜欢能够使用实际位置自动标记照片或找到附近的朋友，但他们也希望选择禁用这些功能。

# 仅在 APP 明确需要时才请求个人数据

对个人信息的请求感到怀疑是很自然的，尤其是在没有明显需求的情况下。确保仅在用户使用明显需要个人数据的功能时才发出许可请求。例如，应用可能仅在激活位置跟踪功能时才请求访问当前位置。

# 解释 APP 为何需要该信息

提供自定义文本（目的字符串或用法描述字符串）以显示在系统的权限请求提示中，并提供示例。保持文字简短明了，使用句子大写并保持礼貌，让用户不会感到压力。无需包括您的 APP 名称 - 系统已经可以识别您的应用程序。有关开发人员的指导，请参阅保护用户隐私。

# 仅在 APP 运行时才在启动时请求权限

如果很明显您的应用程序依赖于其个人信息进行操作，那么用户将不会对此请求感到困扰。

# 不要不必要地请求位置信息

在访问位置信息之前，请检查系统查看是否启用了位置服务。有了这些知识，您可以将提示延迟到某个功能真正需要的时候，或者完全避免提示。要了解如何实现位置功能，请参见 MapKit 和位置和地图编程指南。

# 使用系统提供的警报

您可以在标准权限提示中自定义文本，但要避免添加自定义提示来复制标准提示的行为或外观。

# 设置

某些 APP 可能需要提供一种方法来进行设置或配置选择，但大多数 APP 可以避免或推迟这样做。成功的应用程序立即适用于大多数人，同时还提供了一些便捷的方式来调整体验。当您将 APP 设计为可以满足大多数人期望的方式运行时，您会减少对设置的需求。

# 推断从系统从能获得什么

如果您需要有关用户，设备或环境的信息，请在可能的情况下向系统查询，而不是询问用户。例如，不要要求某人输入他们的邮政编码，以便您可以显示本地选项，而要寻求使用其当前位置的权限。如果用户拒绝访问其信息，则可以正常地退回到手动输入。

# 周全地考虑 APP 的设置选项的优先级。

APP 的主页面是一些必不可少或经常更改的选项的地方。其他页面最好用于仅偶尔更改的选项。

# 在 “设置” 中公开不经常更改的配置选项

“设置” 应用是在整个系统中进行配置更改的中心位置，但是用户必须离开 APP 才能到达那里。直接在 APP 内调整设置要方便得多。如果您必须提供很少需要更改的设置，请参阅《偏好和设置编程指南》中的 “实现 iOS 设置捆绑包” 以获取开发人员指南。
